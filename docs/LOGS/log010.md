# ファイル管理システム改善・進捗記録

## 概要

Media管理システムを即時アップロード + 論理削除システム + 孤立ファイル対策に改善し、ファイル管理の安全性と復旧可能性を向上させる。

## ✅ 完了済み実装

### Phase 1: 即時アップロード復活（完了）
- ArticleFormを`mode="immediate"`に変更
- バッチアップロード処理削除
- 即座プレビュー復活によるUX大幅改善

### Phase 2: 孤立ファイル検出・削除システム（完了）
- `app/actions/cleanup-actions.ts`実装
  - `getAllStorageFiles()` - 3コンテナ一覧取得
  - `detectOrphanFiles()` - DB差分検出（論理削除ファイル除外）
  - `cleanupOrphanFiles()` - 一括削除
  - `getDeletionStats()` - 統計情報取得
- `/admin/media/cleanup`管理画面実装
  - ストレージ統計ダッシュボード
  - 孤立ファイル検出・削除機能

### Phase 3: 論理削除システム（完了）
- Prismaスキーマ拡張
  ```sql
  deletedAt           DateTime?  -- 削除日時
  deletedBy           String?    -- 削除者のユーザーID  
  scheduledDeletionAt DateTime?  -- 物理削除予定日時（30日後）
  ```
- `app/actions/media-actions.ts`拡張
  - `deleteMediaFile()` - 論理削除実行（30日猶予期間）
  - `bulkDeleteMediaFiles()` - 一括論理削除
  - `restoreMediaFile()` - 削除済みファイル復旧
  - `cleanupExpiredFiles()` - 30日経過ファイル物理削除
  - `getDeletedMediaFiles()` - ゴミ箱機能用
- `/admin/media/cleanup`に復旧機能追加
  - 削除済みファイル一覧表示
  - 復旧ボタン
  - 期限切れファイル物理削除

### 孤立ファイル検出ロジック修正
**重要バグ修正**: 論理削除ファイルが孤立ファイルとして誤検出される問題を解決
- **修正前**: `deletedAt: null`のファイルのみとストレージを比較
- **修正後**: 全DBファイル（論理削除含む）とストレージを比較

### ファイル操作フロー
1. **アップロード**: 即時ストレージ保存 + DB記録
2. **削除**: 論理削除（30日猶予期間）
3. **復旧**: 論理削除フラグ解除
4. **物理削除**: 期限切れファイルの完全削除

1. **画像表示問題修正**
   - 問題: 本番環境で画像プレビューに `undefined` が表示
   - 原因: `NEXT_PUBLIC_STORAGE_URL` 環境変数未設定
   - 解決: `.env.production` に `NEXT_PUBLIC_STORAGE_URL=https://altee.me/api/files` 追加

2. **記事作成での画像削除エラー修正**
   - 問題: 画像削除時にServer Components renderエラー
   - 原因: ストレージキー解析ロジックの不備
   - 解決: `deleteImageAction` でストレージキーを正しく分割
   ```typescript
   // 修正後: 専用コンテナ対応
   const [container, ...keyParts] = storageKey.split('/')
   bucket = container // "article-thumbnails"
   objectKey = keyParts.join('/') // "2024/12/file.webp"
   ```

3. **5分テスト設定（完了）**
   - 論理削除猶予期間: 30日 → 5分（テスト用）
   - UI表示も対応更新
   - 本番環境での動作確認完了

### 3. 画像削除時バリデーションエラー問題
**現状**: 記事作成で画像削除時に必須項目のバリデーションエラーが一瞬表示される
- **機能への影響**: なし（削除・保存は正常動作）
- **根本原因**: ImageUploaderの状態変更がArticleFormの再レンダリングを引き起こし、react-hook-formがバリデーション実行
- **検討した解決策**: 
  1. フォーム状態分離 → 複雑化
  2. バリデーションタイミング制御 → 一時的効果
  3. 動的resolverによるバリデーション無効化 → 効果なし
- **判断**: 機能的問題なし、UX上の軽微な問題として後回し
